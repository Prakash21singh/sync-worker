# AWS S3 Provider

## Overview

The AWS S3 adapter enables file discovery and migration to/from Amazon S3 buckets. Unlike Google Drive and Dropbox, S3 is a flat object store — there is no native folder hierarchy. Folders are simulated using key prefixes (e.g. `documents/reports/file.pdf`).

**Adapter Type:** `AWS_S3`  
**Auth Method:** API Key (Access Key ID + Secret Access Key)  
**Implements:** `BaseStorageAdapter` (not `FolderSupportingAdapter` — S3 has no real folders)  
**SDK:** AWS SDK v3 (`@aws-sdk/client-s3`)

---

## Authentication

S3 does not use OAuth. Credentials are stored per `Adapter` record in the database and passed directly to the SDK client.

| Field             | Description                   |
| ----------------- | ----------------------------- |
| `accessKeyId`     | AWS Access Key ID             |
| `accessKeySecret` | AWS Secret Access Key         |
| `region`          | AWS region (e.g. `us-east-1`) |

There is no token rotation — API keys do not expire unless rotated manually in AWS IAM.

> ⚠️ **Security note:** Follow the principle of least privilege. The IAM user or role associated with these credentials should only have `s3:GetObject`, `s3:PutObject`, and `s3:ListBucket` permissions on the specific bucket being migrated.

**No environment variables required** for S3 — all credentials are passed through the database `Adapter` record at runtime.

## File Listing

Files are listed using `ListObjectsV2Command` from the AWS SDK. The adapter lists all objects in the configured bucket and normalizes them.

```typescript
// SDK call
new ListObjectsV2Command({
  Bucket: bucketName,
  Prefix: prefix, // optional, for sub-folder scoping
});
```

Normalization is handled by `normalizeS3Objects()` in `utils/mapping.ts`. Object keys are used directly as paths.

**Returned shape (normalized):**

```typescript
{
  id: string; // Internal DB ID
  sourceId: string; // S3 object key (e.g. "documents/report.pdf")
  name: string; // Derived from key via normalizeName()
  type: 'FILE' | 'FOLDER';
  mimeType: string | null; // Not returned by S3; must be inferred
  size: number | null;
  path: string | null; // Same as sourceId (the full S3 key)
  migrationId: string;
}
```

## Folder Simulation in S3

S3 does not have a real folder concept. Folders are represented as key prefixes:

```
documents/               ← "folder" (a zero-byte object or just a prefix)
documents/report.pdf     ← file with key "documents/report.pdf"
documents/q1/data.csv    ← file nested two levels deep
```

When listing, keys ending with `/` are treated as `FOLDER` entries. All other keys are `FILE` entries. This is handled in `normalizeS3Objects()`.

> **Migration note:** When migrating _from_ S3 to a hierarchical provider (Google Drive, Dropbox), the adapter must reconstruct folder structure from key prefixes and create the corresponding folders in the destination before uploading files.

## Pending Implementation: downloadFile & uploadFile

These methods are the immediate priority for completing the S3 integration.

### downloadFile (to be implemented)

Should use `GetObjectCommand` from the AWS SDK:

```typescript
import { GetObjectCommand } from '@aws-sdk/client-s3';

const response = await s3Client.send(
  new GetObjectCommand({
    Bucket: bucketName,
    Key: objectKey,
  }),
);

// Stream to Uint8Array
const stream = response.Body as Readable;
```

### uploadFile (to be implemented)

Should use `PutObjectCommand` for standard files. For large files, `@aws-sdk/lib-storage`'s `Upload` class handles multipart automatically:

```typescript
import { Upload } from '@aws-sdk/lib-storage';

const upload = new Upload({
  client: s3Client,
  params: {
    Bucket: destinationBucket,
    Key: destinationKey,
    Body: fileData,
    ContentType: mimeType,
  },
});

await upload.done();
```

> Using `Upload` from `@aws-sdk/lib-storage` is strongly preferred over `PutObjectCommand` — it automatically handles multipart upload for files over 5MB, which is critical for large file migrations.

## Error Handling

> ⚠️ **Not yet implemented.** The current S3 adapter has no retry logic or structured error handling.

The following should be added to match the pattern used by Google Drive and Dropbox adapters:

| Error                             | Recommended Behavior                                    |
| --------------------------------- | ------------------------------------------------------- |
| `SlowDown` / `ServiceUnavailable` | Retry with `retryWithBackoff` (4 attempts, 500ms base)  |
| `NoSuchKey`                       | Treat as file not found, mark `MigrationFile` as FAILED |
| `NoSuchBucket`                    | Fail the entire migration job immediately               |
| `AccessDenied`                    | Fail immediately, surface clear IAM permission error    |
| `RequestTimeout`                  | Retry with backoff                                      |

## Known Gaps & Future Work

- **`downloadFile` and `uploadFile` are not implemented** — this is the critical blocker for S3 migrations.
- **No retry logic** — needs `retryWithBackoff` added to match other adapters.
- **No error handling** — structured error handling for AWS SDK errors is missing.
- **MIME type detection** — S3 does not return MIME types in list responses; consider using the `mime-types` npm package to infer from file extensions.
- **`buildDownloadRequest` and `buildUploadRequest`** optional interface methods are not implemented; add them to plug S3 cleanly into the migration worker.
- **Multipart uploads** — use `@aws-sdk/lib-storage` `Upload` class rather than `PutObjectCommand` to handle files above 5MB reliably.
- **S3 to hierarchical provider migrations** require folder reconstruction logic from key prefixes.
